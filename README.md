## 🏗️ Что это такое?

Частично распределенная консольная платформа для централизованного управления несколькими торговыми депозитами (или субаккаунтами) на централизованных-же криптобиржах. Развертывание предполагается исключительно в среде Linux, можно теоретически в контейнерах Docker. 

### Ключевые особенности

**Архитектура:**
- Консольное PHP приложение с циклом ~1 минута
- MariaDB с двухсторонней репликацией основной БД tradimg для хранения заявок, сигналов, конфигурации
- REST API для взаимодействия с биржами
- Простой REST API для получения внешних сигналов с вашего сервера
- Распределенность через параллельный запуск двух экземпляров, дающая относительную отказоустойчивость

**Функциональность:**
- **Плавное исполнение**: типично 5-10 минут на полное заполнение позиции
- **Оптимизация проскальзывания**: пошаговое исполнение с адаптацией
- **Market Making (тестовый)**: размещение заявок на нескольких уровнях
- **Сеточная торговля**: примитивная реализация с фиксированным шагом
- **Батчинг ордеров**: группировка связанных заявок для отслеживания

**Поддерживаемые биржи:**
- Binance
- Bitfinex  
- BitMEX
- Deribit
- Bybit

## 🏗️ Архитектура

### Принцип работы

```
┌─────────────────────────────────────────────────────────────┐
│  TradingCore::Run()  — главный цикл (~1 мин интервал)       │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ TradingCore::Update()  — синхронизация с биржей       │  │
│  │  ├─ Проверка статуса ордеров                          │  │
│  │  ├─ Обновление позиций                                │  │
│  │  ├─ Загрузка внешних сигналов (REST API)              │  │
│  │  └─ Запись в БД (MariaDB с репликацией)               │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ TradingCore::Trade()  — расчет и выставление заявок   │  │
│  │  ├─ CalcTargetPos() — расчет целевых позиций          │  │
│  │  ├─ ExtSigVerify() — валидация внешних сигналов       │  │
│  │  ├─ SetupLimits() — настройка ограничений             │  │
│  │  └─ NewOrder() — размещение на бирже                  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**⏱️ Хардкод цикла:** Интервал 1 минута **не настраивается**, так как влияет на логику. На практике с некоторыми биржами и большим числом позиций одной минуты недостаточно для полной синхронизации, тогда обычно следует пропуск цикла.

### Модульная структура (трейты)

Проект разбит на специализированные трейты:

```
TradingCore (ядро)
├── TradingLoop      # Торговый цикл и логика исполнения
├── TradeLogging     # Структурированное логирование
├── MarketMaker      # Маркет-мейкинг (экспериментальный)
└── Reporting        # Отчетность и статистика

### Ключевые компоненты

#### 1. **TradingCore** (`trading_core.php`)
Центральное ядро системы. Управляет жизненным циклом бота:
- Инициализация подключений к биржам
- Загрузка конфигурации из БД
- Координация работы всех модулей
- Обработка событий и ошибок

#### 2. **TradingLoop** (`trading_loop.php`)
Реализует основной торговый цикл через метод `Trade()`:
```php
1. ScanForTrade()    — получение целевых позиций из БД
2. CalcTargetPos()   — расчет требуемых изменений с учетом коэффициентов
3. SetupLimits()     — настройка BoundedValue ограничителей
4. ExtSigVerify()    — верификация и приоритизация внешних сигналов
5. Trade()           — размещение ордеров с контролем рисков
```

**Особенность:** Плавное исполнение за несколько циклов (5-10 минут), а не мгновенное. По сути бот двигает заявки все ближе к спреду, и только в режиме спешки внутри спреда пытается провести исполнение. На неликвиде это все никак неизбавляет от большого проскальзывания, при резких движениях рынка оно может даже возрастать.

#### 3. **TradingContext** (`trading_context.php`)
Контейнер состояния для каждой торговой пары с умными переменными:
```php
class TradingContext {
    private array $vars = []; // BoundedValue переменные
    
    // Основные параметры:
    // - curr_pos: текущая позиция
    // - target_pos: целевая позиция (с zero_cross защитой)
    // - bias: абсолютное смещение (unsigned)
    // - amount: объем ордера (с max_abs)
    // - cost_limit: лимит стоимости
}
```

#### 4. **BoundedValue** (`lib/smart_vars.php`)
Умные переменные с декларативными ограничениями:
```php
$ctx->get_var('target_pos')
    ->constrain('zero_cross', 0)      // Запрет пересечения нуля
    ->constrain('max_abs', $max_pos)  // Лимит позиции
    ->onChange(function($new, $old) {
        // Реакция на изменение
    });
```

**Поддерживаемые ограничители:**
- `max_abs` — максимальное абсолютное значение
- `max/min` — диапазон значений
- `multiple_of` — кратность (для lot_size)
- `zero_cross` — запрет смены знака
- `callback` — пользовательская проверка

#### 5. **OrdersBatch** (`orders_batch.php`)
Группировка связанных ордеров:
- Отслеживание прогресса исполнения
- Автоматическое управление lifecycle
- Связь с внешними сигналами
- Статистика исполнения

#### 6. **ExternalSignal** (`ext_signals.php`)
Обработка торговых сигналов:
- Расчет дельты позиции
- Управление приоритетами
- Балансировка заполнения
#### 7. **MarketMaker** (`market_maker.php`)
**⚠️ Экспериментальный модуль:**
- Размещение заявок на нескольких уровнях от спреда
- Примитивная сеточная торговля (Grid) с фиксированным шагом
- Работает с внешними сигналами класса `GridSignal`

### 8. Signals Server (Legacy)

**⚠️ Legacy-компонент** — автономный PHP веб-сервер для управления сигналами:
- `get_signals.php` — REST API для получения сигналов
- `signal_data.php` — хранение и обработка сигнальных данных
- `sig_import.php` — импорт сигналов (TradingView-strategy)
- `lastpos.php` — отслеживание текущих позиций
- `health_svg.php` — мониторинг состояния
- Telegram Bot интеграция

### Уровень 1: Приложение (TradingCore)

**`TradingCore`** — самый перегруженный функциональностью класс, по сути реализующий всё приложение через композицию трейтов.

```
┌─────────────────────────────────────────┐
│          TradingCore                    │  ← Базовый класс приложения
│  (композиция функциональности)          │
├─────────────────────────────────────────┤
│  use TradingLoop;      // Торговля      │
│  use TradeLogging;     // Логирование   │
│  use MarketMaker;      // ММ (опц.)     │
│  use Reporting;        // Отчеты        │
├─────────────────────────────────────────┤
│  + Run()           // Главный цикл      │
│  + Update()        // Синхронизация     │
│  + Trade()         // Торговая логика   │
│  + LoadConfig()    // Конфигурация      │
└─────────────────────────────────────────┘
           ▲
           │
     ┌──────┴───────┬──────────────┬──────────────┬──────────────┐
     │              │              │              │              │
 ┌───┴────┐    ┌────┴─────┐   ┌────┴─────┐   ┌────┴─────┐   ┌────┴─────┐
 │Binance │    │ Bitfinex │   │  BitMEX  │   │ Deribit  │   │  Bybit   │

 └────────┘    └──────────┘   └──────────┘   └──────────┘   └──────────┘
      ↓              ↓            ↓             ↓             ↓
    impl_       impl_        impl_         impl_          impl_
  binance.php bitfinex.php bitmex.php  deribit.php    bybit.php
```

**Наследники `TradingCore`** (impl_*.php):
- Определяют **конкретные инстансы ботов** для каждой биржи
- Задают `impl_name` (например, `'binance'`, `'deribit'`)
- Создают и инициализируют объект **движка** для конкретной биржи
- **НЕ добавляют** особой функциональности — вся логика в `TradingCore`

```php
// Пример: impl_binance.php
class BinanceTrading extends TradingCore {
    
    public function __construct($mysqli) {
        parent::__construct($mysqli);
        
        $this->impl_name = 'binance';  // Идентификатор
        
        // Создание движка для Binance
        $this->trade_engine = new BinanceEngine($this);
    }
}
```

### Архитектурная цель

```
Приложение (TradingCore)
    ↓ зависит от
Движок (TradingEngine/RestAPIEngine)
    ↓ абстрагирует
Биржа (Binance/BitMEX/Deribit API)
```

**Принцип разделения:**
1. **TradingCore** — бизнес-логика торговли, единая для всех бирж
2. **RestAPIEngine** — общие паттерны REST API (HTTP, retry, rate limiting)
3. **BinanceEngine/BitMEXEngine/...** — специфика конкретной биржи

**Результат:** Добавление новой биржи = реализация одного класса движка, без изменений в `TradingCore`.

### Конвейер доставки API Secret до подписи запроса

Ниже описан типовой путь секрета, который считается эталонным для текущей архитектуры.

1. **Источник секрета**
    - `bot_manager.php` получает API Secret из БД (`loadApiFromDb`) или из pass (`loadApiFromPass`).
    - На этом этапе секрет находится в plaintext только в памяти процесса bot_manager.

2. **Передача в процесс бота через ENV**
    - Перед запуском `bot_instance.php` менеджер кладет secret в ENV в base64-форме:
      - `BMX_API_SECRET` / `BYBIT_API_SECRET` / ...
      - служебный флаг `__CREDS_FROM_ENV=1`.
    - Формат передачи: `base64(plaintext_secret)`.

3. **Инициализация в движке**
    - `RestAPIEngine::InitializeAPIKey()` читает key/secret из ENV или runtime-файлов.
    - Важный инвариант: метод не должен выполнять обязательный decode секрета «за все биржи».
    - Причина: каждая биржевая реализация имеет собственный контракт того, в каком виде она ожидает secret перед подписью.

4. **Подготовка секрета в конкретной бирже**
    - BitMEX: `impl_bitmex.php::SignRequest()` собирает secret и декодирует из base64 перед HMAC.
    - Bybit: `impl_bybit.php::DecodeSecret()` декодирует packed/base64, затем `SignRequest()` использует raw-байты.
    - Bitfinex/Deribit: выполняют декодирование на своем этапе подписи.

5. **Метод подписи**
    - Подпись строится только в exchange-специфичном методе (`SignRequest`/аналог).
    - Именно этот уровень считается единой точкой правды для decode + HMAC-алгоритма биржи.

#### Критические инварианты (чтобы не словить двойную развертку)

- **Decode секрета должен происходить ровно один раз на конкретный запрос подписи**.
- **Глобальный слой (`InitializeAPIKey`) не должен принудительно декодировать секрет для всех движков**.
- **Если меняется формат доставки секрета из bot_manager, надо синхронно проверить все `SignRequest` реализации**.
- **Нельзя смешивать два источника одновременно (ENV + runtime-file) без явного приоритета и логирования источника**.

#### Что проверять при правках

1. В логах движка видно корректный профиль/API base, инициализация проходит без auth-ошибок формата signature mismatch.
2. `CheckAPIKeyRights()` проходит для валидного ключа и возвращает ожидаемые права (`order` и т.д.).
3. Нет повторного decode в двух слоях подряд (общий слой + биржевой слой).
4. Любое изменение в credential pipeline сопровождается smoke-проверкой хотя бы для BitMEX и Bybit.

## 🚀 Развертывание

**Развертывание платформы — нетривиальный процесс**, требующий:
- Загрузка зависимых компонентов с других моих репозиториев
- Конфигурации MariaDB репликации
- Настройки хостов на системном уровне в /etc/hosts, чтобы не изменять код

- По сути требуется два локальных хоста, в идеале с независимым интернетом, и один удаленный для сервера сигналов и вспомогательных компонентов. 

**Документация по установке будет создана и выделена в отдельный `INSTALL.md`** только по активным запросам желающих изучить платформу.

- **PHP**: 8.1+
- **MariaDB**: 10.5+ 
- **Расширения PHP**: 
  - `mysqli`, `curl`, `json`, `pcntl`
  - `gmp` (для BitMEX HMAC подписей)

### Конфигурация

#### 1. Реквизиты БД задаются в `lib/db_config.php`
#### 2. Секретная часть ключа API обычно делиться на три части, две сохраняются через приложение pass в отдельные файлы, а разделитель в конфигурацию (БД). Экземпляр бота может считывать из переменных окружения сам ключ и его секрет. Если запускать ботов через bot_manager.php, ключи будут передаваться автоматически, при условии разблокированной БД паролей. 

```bash
    # Отладочный запуск бота для конкретной биржи
    php bot_instance.php binance_bot

    # С логированием в файл
    php bot_instance.php binance_bot > logs/stdout.log 2> logs/stderr.log       

    # в продакшен режим, через менеджер - все боты будут работать до полуночи UTC, и автоматически перезапускаться. 
    ./bot_manager.php
```

Платформа использует БД trading для всех настроек, где каждому боту предназначена таблица с префиксом config__, а все таблицы перечислены в config__table_map. 

**Распределенность:** При параллельном запуске двух экземпляров один автоматически становится резервирующим (rescuer role), обеспечивая отказоустойчивость. Подразумевается две связанные репликацией локальные БД, через которые экземпляры могут оценивать активность друг друга, для чего используется таблица bot__redudancy.

$ctx->get_var('target_pos')
    ->constrain('zero_cross', 0)           // Не менять long↔short
    ->constrain('max_abs', $this->max_pos); // Макс. позиция

// Ограничение объема ордера
$ctx->get_var('amount')
    ->constrain('min', $ti->lot_size)       // Минимальный лот

    ->constrain('min', $ti->cost_limit)

## 📝 Логирование

### Цветные логи

Система использует escape-коды для раскрашивания:

```php
$this->LogMsg("~C92#SUCCESS:~C00 Order filled %s", $order_id);
$this->LogError("~C91#ERROR:~C00 API call failed: %s", $error);
$this->LogMsg("~C93#DEBUG:~C00 Position: %f", $position);
```

**Коды цветов:**
- `~C00` — сброс
- `~C91` — красный (ошибки)
- `~C92` — зеленый (успех)
- `~C93` — желтый (debug)
- `~C94` — синий (info)
- `~C95` — магента (важное)

### Структура логов

```
logs.td/
├── binance_YYYY-MM-DD.log    # Основные логи
├── errors_YYYY-MM-DD.log     # Только ошибки
├── orders_YYYY-MM-DD.log     # История ордеров
└── positions_YYYY-MM-DD.log  # Изменения позиций
```

Журналы в случае активного трейдинга, или повторяющихся сбоев могут очень активно расти, вплоть до нескольких гигабайт в сутки. Поэтому реализована автоматическая компрессия оных, при обработке штатного завершения приложения. В любом случае нужно с запасом выделить дисковое пространство.

Управление через Telegram-бот реализовано в `tele-bot/trade_ctrl_bot.php`.

**Архитектура:**
```
TradingCore → event_sender.php → trade_ctrl_bot.php → Telegram API
```

- `event_sender.php` — локальный буфер событий (может аккумулировать при недоступности сервера)
- `trade_ctrl_bot.php` — мини-сервер на хостинге, прокси к Telegram Bot API

**Доступные команды:**
- `bot-status` — статус торговых экземпляров
- `restart` — перезапуск бота
- `pos_coef <value>` — изменение коэффициента позиций
- `pos_offset <pair> <offset>` — административное смещение позиции
- `signal` — управление внешними сигналами

Команды защищены системой прав доступа по Telegram User ID.

**Автоматические отчёты в канал:**
- Каждые 10 минут: отчёт о крупных заявках и группах мелких ордеров

- Два раза в день: состояние счета с графиком Equity

## 🏁 Заключение 

Этот проект — пример того, **насколько сложными могут быть системы автоматической торговли**, когда необходимы:
- Высокая надежность и отказоустойчивость
- Контроль рисков на каждом уровне
- Поддержка множественных бирж и стратегий
- Распределенная архитектура

В то же время, это пример того, как **архитектура может зайти в тупик** под давлением убытками и системно несерьёзном контроле за финансовой стороной. 
Важным провалом является отсутствие документации хода разработки, журнала сбоев и баг-фиксов, даже непосредственно приватного репозитория проекта за 4 года не существовало. Если вы захотите реализовать свою платформу, просчитайте заранее пирамиду фокуса: контроль рисков, оценка сделок с ведением статистики P/L, эффективная отчетность, автоматика сворачивания торговли при пороговых убытках. Собственно я о рисках, особенно коварных при маржинальном трейдинге, начал задумывать лишь последний год, но в финансовом плане уже оказалось поздно. Целиком сосредоточившись на ядре и технических аспектах, я сильно упустил момент когда торговля внезапно давать убытки в 5-7% за день, а срезать риски быстро не захотел. Автоматизировано уменьшение торгового коэффициента было например в феврале 2025, через 4 года от создания первых исходников. Основа финансового успеха в трейдинге, это навык сбережения депозита, а не сверхвысокая доходность (и как следствие риск). По минимуму вы должны иметь финансовый стоп, после которого торговля останавливатся вся и до полного разбора ситуации, исправления и проведения тестов.

Используйте этот код как **учебный материал**, но не как готовое решение. Написать что-то подобное в режиме итеративной эволюции проекта, с помощью команды программистов вероятно будет стоить сотни тысяч долларов, т.е. многое можно оценить во избежание многомесячной эпопеи. Причем скорость движения к результату при живой торговле будет очень медленной, так что лучше реализовать поддержку демо-трейдинга и оценить нагрузочную способность вашего решения в разных состояниях рынка и при различных искусственных сбоях.
