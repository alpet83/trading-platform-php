=============== ОПТИМИЗАТОР ИСПОЛНЕНИЯ (КОНЦЕПТ). ============================
 Отдельный скрипт каждую минуту обрабатывает поток минутных свечей, делая маркировки для каждого ТФ: направление движения цены, факт расширения диапазона. 
 Исполняющие скрипты соответственно используют такую маркировку, при условии актуализированных данных. Если диапазон расширяется и есть движение цены в сторону открытия позиции,
 используется наиболее агрессивный выбор цены. Если диапазон не расширяется, или цена движется супротив открытия заявки, используется спокойный выбор цены.     
 В зависимости от стратегии ленивого или быстрого исполнения, производится разбивка крупных входов на части - с расчетом на поочередное исполнение небольшими объемами.
 
 >>> Результатами обработки ценовых данных, является:
   1.  целочисленный фактор скорости (от 0 до 15) исполнения
   2.  максимальный объем заявки, что ограничивает % выполнения сигнала в минуту
   3.  худшая возможная цена выставления заявки
   4.  прогнозное время отработки сигнала в условиях оптимизации
   
 
 
 
 Блокиратор: в случае экстремальных показаний RSI 1h цену открытия лимитировать соотвествующей последнему значению RSI в пределах "нормы" (30-60), допуская продление исполнения входа 
 до закрытия следующего часа. После второго часа в перепроданности, разрешать вход очень малыми объемами (100$ в заявке, в минуту). Отмена блокировки при расширении недельного диапазона.

======================= СЕТОЧНОЕ ИСПОЛНЕНИЕ ==========================================
Концепция ордерблока: набор заявок с одним направлением, равным объемом, и распределением через равные интервалы.
В режиме ожидания бот может держать два ордерблока на большой дистанции от спреда, уподобляясь маркет-мейкеру. Когда нужно открывать/закрывать позицию, он активизирует третий ордерблок, 
в который переносит заявки из соответствующего (аски или биды), при этом интервал сжимается до уровня локальной волатильности 10-15 минут. Наполнение исполняющего ордерблока должно 
зависеть от целевого объема: от 1 до 5 заявок с фиксированным объемом, где последняя соедржит остаток. Выставление заявок внутрь спреда предполагает полное скрытие объема (айсберг). 
* Если спред очень большой, можно выставлять провокационные противофазные заявки с минимальным объемом, чтобы появился оригинальный ММ. 
  
  
====================== СЕТОЧНЫЕ БОТЫ =================================================
Первая реализация получилась очень простой:
 1. Заданный диапазон цены делиться на заданное количество шагов, получаются уровни сетки.
 2. На каждом уровне, в зависимости от текущей цены спреда может быть выставлена лимитка на продажу или покупку
 3. Уровень сетки освобождается, когда по ближнему к нему заявка исполняется с противоположным целевому направлению.
Такая реализация может последовательно набирать и разгружать позицию, которая всякий раз имеет пресказуемое значение на каждом уровне. 
Чтобы тейкинг происходил реже, но всегда только по прибыльному набору, можно попробовать использовать среднюю цену набора, как фильтр. Такой подход впрочем чаще всего должен 
блокировать работу сеточного бота. Альтернативой является, при заметном наборе (несколько уровней) позиции - тейкать немного больше настроечного лота. Однако тогда очистка 
исполненных заявок взаимным коллапсом, вероятно даст отмену закрытия в масштабе общей позиции. Скажем когда большой лонг набран (во время обвала), тейки продажи на падении увеличенные,
и коллапс большой продажи с обычным откупом, убирает приращение из расчета текущей дельта-позиции и лонг будет восстановлен доп. заявкой. 
 
  
====================== АРБИТРАЖ И РАСПРЕДЕЛЕНИЕ ЛИКВИДНОСТИ ==========================
1. Нужно реализовать анализ ликвидности, через оценку объемов проходящих по каждой паре, наличию крупных заявок около спреда (проверка стакана), с выводом универсального индекса
2. При активированном арбитраже, поступающий сигнал распределяется арбитром на сервере, так чтобы максимальный объем был исполнен там где меньше всего будет проскальзывание. 
   Далее происходит распределение объема между биржами, небольшими лотами, т.е. скажем крупный закуп распродается, и на остальных биржах происходит откуп. 
3. Механизм может включать аккаунты-мулы, где объем будет набираться исключительно для распределения ликвидности, но не будет удерживаться долго. Например основные депозиты могут 
быть спотовые, в то время как мулы будут использовать маржинальные пары (там где спот неликвиден, например BitMEX).

-- Table disk usage
 SELECT table, formatReadableSize(size) as size, rows, days, formatReadableSize(avgDaySize) as avgDaySize FROM (
    SELECT
        table,
        sum(bytes) AS size,
        sum(rows) AS rows,
        min(min_time) AS min_time,
        max(max_time) AS max_time,
        toUInt32((max_time - min_time) / 86400) AS days,
        size / ((max_time - min_time) / 86400) AS avgDaySize
    FROM system.parts
    WHERE active
    GROUP BY table
